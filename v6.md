# TECH SPEC V6 : Trading Bot (Alpaca / TypeScript / Bun)

## 1. Contexte & Gestion du Calendrier Boursier

Mise à jour majeure du bot algorithmique visant à sécuriser l'infrastructure, journaliser les métriques quantitatives et adapter dynamiquement les entrées/sorties.

### RÈGLE D'OR 1 : Gestion des Jours Fermés (Week-end & Jours Fériés)

Le bot doit impérativement vérifier l'état du marché avant d'initier la moindre tâche planifiée (Screener, Pré-market, etc.).

- **Vérification Calendrier :** Utiliser l'endpoint de l'API Alpaca (`GET /v2/calendar`) pour valider si le jour actuel est un jour de trading ouvert.
- **Mode Week-end / Fermé :** Si le marché est fermé et qu'aucune session n'est planifiée, le bot bascule en mode veille. **Interdiction stricte d'ouvrir ou de maintenir les WebSockets (WS) actifs.** Les connexions WS existantes doivent être explicitement fermées de manière propre pour éviter les erreurs de reconnexion en boucle (Code 406 / Limit Exceeded) constatées les week-ends.

### RÈGLE D'OR 2 : Déduplication Absolue de l'Univers (Core vs Satellite)

Il est strictement interdit qu'un même symbole réside simultanément dans la liste Core et dans la liste Satellite lors d'une même session.

- **Logique de Priorité :** La Watchlist Core (`V1_CORE`) a la priorité absolue.
- **Filtrage dans `premarket_screener.ts` :** Lors de l'exécution du screener Play-Maker V2 en pré-market, le script doit charger la liste des symboles déjà retenus par le screener Core. Tout candidat V2 dont le symbole est déjà présent dans la liste Core doit être **immédiatement rejeté** de la sélection Satellite, quel que soit son score ou son gap.

## 3. Modules

### 3.1 `notificationManager.ts`

Service asynchrone non bloquant (try/catch global).

- **Endpoint :** `POST https://api.telegram.org/bot<TOKEN>/sendMessage`
- **Payload :** `chat_id`, `text`, `parse_mode: 'HTML'`
- **Triggers :** démarrage (09:30 EST), watchlist Core/Satellite, exécutions d'ordres, rapport post-mortem (16:00 EST).

### 3.2 `journalManager.ts`

Interface `TradeRecord` persistée en JSON local (`data/journal.json`) ou Prisma si disponible.

| Phase      | Champs                                                                        |
| ---------- | ----------------------------------------------------------------------------- |
| Pre-Trade  | `symbol`, `origin` (Core/Satellite), `alpha_vs_spy`, `relative_volume`        |
| Exécution  | `entry_time`, `entry_price`, `vwap_at_entry`, `ema9_at_entry`, `spy_trend_5m` |
| Résolution | `exit_time`, `exit_price`, `net_pnl_dollars`, `exit_reason`                   |
| Quant      | `MFE_percent`, `MAE_percent`                                                  |

**Capture :** initialisation à l'achat (`trader.ts`), complétion à la vente, push dans le journal.

### 3.3 `analyzer.ts`

Post-mortem EOD à 16:00 EST : Win Rate, ratio Average Win/Loss, résumé formaté envoyé via Telegram. Sauté si aucun trade n'a été exécuté.

### 3.4 `riskManager.ts`

- **Daily Kill-Switch :** si Net PnL ≤ -$1,500 (-1.5%) → statut `HALTED`, liquidation au marché, annulation des ordres, blocage des entrées.
- **EOD Sweep :** au Hard Close (15:58 EST), appeler `alpaca.cancelAllOrders()` **avant** les Market Sell pour éviter une double liquidation (stop latent → position short).

### 3.5 `trader.ts` — Entrée

- **Volume Confirmation (Satellites) :** sur signal rebond VWAP/EMA9, l'achat ne s'exécute que si le volume de la bougie de cassure (1-min) > VMA_10 (Moyenne des volumes des 10 dernières bougies).

### 3.6 `positionManager.ts` — Sortie dynamique

Évaluée à chaque tick WebSocket ou nouvelle bougie 1-min, **avant** les cibles fixes (5% / 7%).

#### A. ATR Dynamic TP & SL

Niveaux initiaux basés sur la volatilité 5-min :

- Target 1 = `1.5 × ATR(5m)`
- Stop Loss = `1.0 × ATR(5m)`

#### B. Time-Decay (EST)

| Fenêtre       | Comportement                                            |
| ------------- | ------------------------------------------------------- |
| Avant 13:00   | Cibles normales                                         |
| 13:00 – 14:30 | Cibles TP divisées par 2                                |
| Après 14:30   | Pas de cible fixe ; si PnL > +1.5% → trailing stop 0.5% |

#### C. Smart Exit — RSI Overbought

- **Condition :** PnL > 1% **et** RSI(14, 1-min) > 80
- **Action :** Market Sell intégral. Log : `RSI_OVERBOUGHT_EXIT`

#### D. Smart Exit — Volume Exhaustion

- **Condition :** PnL > 0.5% **et** volume < 50% du VMA_10 sur **2 bougies consécutives**
- **Action :** trailing stop serré 0.2%. Log : `VOLUME_EXHAUSTION_TRAILING`
